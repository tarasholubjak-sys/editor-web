# Skill: Selfy Knowledge Editor

Переробляє сирі робочі документи компанії Selfy (Google Docs, .docx, тексти з чатів, нотатки) у **стандартизовані статті** для Outline-бази знань.

## Коли активувати

Спрацьовує по тригерах у запиті користувача:

- "переписати документ"
- "оформи статтю"
- "стандартизуй цей текст"
- "перероби для бази знань"
- "/refactor"
- Користувач кидає файл/текст + явно або неявно просить його привести до стандарту

## Що робить

1. **Класифікує** тип документа (SOP / Скрипт / FAQ / Онбординг / Політика / Опис продукту).
2. **Витягує суть** з сирого тексту: основні факти, кроки, правила, цитати.
3. **Прибирає шум**: повтори, "воду", хаотичне оформлення, англіцизми/росіянізми.
4. **Застосовує шаблон** відповідного типу (`templates/{type}.md`).
5. **Заповнює стандартну шапку** з метаданими (власник, для кого, дата, статус, теги).
6. **Доповнює відсутнє**:
   - Чек-листи виконання
   - Типові помилки
   - Шаблони фраз (для скриптів)
   - "Як зрозуміти що ок"
7. **Позначає дірки** через секцію `⚠️ Потрібно уточнити` — НЕ вигадує фактів.
8. **Використовує можливості Outline 1.8** (notice blocks, toggle, mermaid, draw.io, document mentions, PDF embeds).
9. **Видає готовий markdown** для копіювання в Outline.

## Структура скіла

```
selfy-knowledge-editor/
├── SKILL.md                       ← цей файл (опис активації)
├── prompt.md                      ← головний промпт-редактор
├── rules/
│   ├── outline-capabilities.md    ← всі фічі Outline 1.8
│   ├── formatting-style.md        ← правила оформлення Selfy
│   └── selfy-context.md           ← контекст компанії
├── templates/
│   ├── sop.md                     ← Регламент/Процес/Інструкція
│   ├── script.md                  ← Скрипт продажу/дзвінків
│   ├── faq.md                     ← FAQ
│   ├── onboarding.md              ← Онбординг новачка
│   ├── policy.md                  ← Політика/Правила
│   └── product.md                 ← Опис продукту/бренду
└── examples/
    ├── return-before.md           ← сирий приклад
    └── return-after.md            ← стандартизований результат
```

## Як використовувати

### Сценарій 1 — у чаті Claude

```
Користувач: [вставляє сирий текст]
            "перепиши це для бази знань"

Claude: → читає SKILL.md, prompt.md, відповідний template, rules/
        → класифікує (SOP)
        → застосовує шаблон
        → видає готовий markdown
```

### Сценарій 2 — у Telegram-боті команда `/refactor`

```
Користувач: /refactor [файл або текст]

Бот: → витягує текст з файлу (gdoc/docx/pdf/md/txt)
     → завантажує prompt.md + rules/ + templates/
     → запускає LLM з повним промптом
     → повертає готовий markdown файлом або в чат
```

### Сценарій 3 — CLI скрипт

```bash
node scripts/refactor-doc.js /path/to/file.docx
# → створює drafts/2026-05-28__SOP__назва.md
```

## Вихід

Готовий markdown-документ, який:

- Має шапку з метаданими (власник, для кого, оновлено, статус, теги)
- НЕ має блоку "Зміст" (Outline сам генерує збоку з H2-H4)
- Структурований за відповідним шаблоном
- Використовує Notice-блоки (`:::info`, `:::warning`, `:::tip`, `:::success`) де доцільно
- Використовує Mermaid діаграми для алгоритмів
- Має чек-листи, типові помилки, "як зрозуміти що ок"
- Позначає невідомі факти в `⚠️ Потрібно уточнити`
- Українською, стисло, без води, для всіх співробітників (не лише менеджерів)

## Критерії готовності

Документ вважається готовим, коли виконано всі пункти:

- [ ] Є шапка з метаданими
- [ ] H1 = назва, далі H2/H3 — без H4 в TOC
- [ ] Емодзі на H2 (один на секцію)
- [ ] Notice-блоки використовуються де доцільно
- [ ] Є хоча б один чек-лист (де доречно)
- [ ] Є "Типові помилки" або "Як перевірити результат"
- [ ] Враховано контекст Selfy (бренди, правила повернення, термінологія)
- [ ] Дірки позначено `⚠️ Потрібно уточнити` замість вигадок
- [ ] Текст придатний до виконання без додаткових пояснень
